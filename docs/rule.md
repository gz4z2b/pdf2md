# 项目规范

本文档规定 pdf2md 项目的开发方式。后续写代码、改功能、修问题时，都应先阅读本规范。

## 基本原则

1. 先满足用户最重要的目标：把 `origin` 目录里的 PDF 批量转换成 Markdown。
2. 优先使用简单、稳定、容易理解的实现方式。
3. 每个功能只做自己该做的事，避免把多个职责混在一个文件里。
4. 用户能直接使用的入口必须尽量简单，v1.0.0 统一使用 `scripts/convert.sh`。
5. 每次新增功能后，都要同步更新 README 和 `docs/functions` 中的功能说明。

## Python 版本

项目使用 Python 3.11 或更高版本。

原因：

1. Python 3.11 性能更好。
2. 类型提示和异常信息更友好。
3. 依赖库兼容性较好。

## 目录规范

```text
src/pdf2md/
├── __init__.py
├── cli.py
├── config.py
├── batch.py
├── pdf_reader.py
├── markdown_writer.py
├── logger.py
└── errors.py
```

| 目录或文件 | 说明 |
| --- | --- |
| `origin/` | 用户放 PDF 原文件的目录 |
| `output/` | Markdown 输出目录，保留原 PDF 的子目录结构 |
| `scripts/` | 一键执行脚本目录 |
| `src/pdf2md/` | 项目正式源代码目录 |
| `tests/` | 自动化测试目录 |
| `docs/functions/` | 每个功能的用户说明 |
| `docs/v1.0.0/` | v1.0.0 需求、方案和问题记录 |

## 分层规范

| 层级 | 负责内容 | 不应该做的事 |
| --- | --- | --- |
| `cli.py` | 解析命令行参数，启动任务 | 不直接解析 PDF |
| `batch.py` | 扫描目录、安排批量转换 | 不关心 PDF 内部结构细节 |
| `pdf_reader.py` | 读取 PDF 页面、文字块、图片信息 | 不负责写文件 |
| `markdown_writer.py` | 生成 Markdown 文本和文件 | 不负责扫描目录 |
| `config.py` | 管理默认路径和可配置参数 | 不写业务逻辑 |
| `errors.py` | 定义项目异常 | 不输出用户日志 |
| `logger.py` | 统一日志格式 | 不处理业务流程 |

## 依赖规范

推荐依赖：

| 依赖 | 用途 |
| --- | --- |
| `pymupdf` | PDF 解析 |
| `pytest` | 测试 |
| `pytest-cov` | 测试覆盖率 |
| `reportlab` | 测试中生成 PDF 样本 |

依赖应写入 `requirements.txt`，不要只写在脚本里。

## 代码风格

1. 文件名、函数名、变量名使用小写英文和下划线，例如 `convert_pdf_to_markdown`。
2. 类名使用大驼峰，例如 `PdfReadError`。
3. 函数应尽量短小，单个函数只处理一个明确任务。
4. 公共函数必须写类型提示。
5. 复杂逻辑需要写简短注释，普通赋值和一眼能看懂的代码不要写废话注释。
6. 面向用户的错误信息要直接说明原因和处理办法。

## 日志规范

程序运行时应输出下面几类信息：

| 日志类型 | 示例 |
| --- | --- |
| 开始转换 | `开始扫描 origin 目录` |
| 文件进度 | `正在转换：origin/demo.pdf` |
| 转换成功 | `已输出：output/demo.md` |
| 转换失败 | `转换失败：文件已损坏或无法读取` |
| 最终汇总 | `完成：成功 9 个，失败 1 个` |

日志默认输出到终端。后续如果需要，也可以扩展输出到日志文件。

## Markdown 输出规范

1. 每个 PDF 输出一个同名 `.md` 文件。
2. 输出目录为 `output`。
3. PDF 在 `origin` 中的子目录结构，应在 `output` 中保留。
4. 页面之间使用二级标题分隔，例如 `## 第 1 页`。
5. 图片内容使用占位符，例如 `[图片占位符：第 1 页第 2 张图片]`。
6. 空白页应写入提示，例如 `[本页没有可识别文字]`。

## 开发流程规范

1. 开发前先阅读 `README.md`、`docs/rule.md`、`docs/test.md` 和对应版本需求。
2. 查找代码、理解逻辑、分析影响面时优先使用 CodeGraph。
3. 如果 CodeGraph 没有索引，先生成索引再继续分析。
4. 写代码前先补测试用例，并让用户确认测试目标。
5. 代码完成后执行相关测试，直到测试通过。
6. 功能完成后更新对应功能文档和版本问题记录。

## 文档维护规范

1. README 只做总入口，不堆放太多细节。
2. 功能细节写在 `docs/functions`。
3. 版本需求、实现方案和问题记录写在对应版本目录。
4. 文档语言要让非技术用户能看懂，避免只写开发者才懂的缩写。
