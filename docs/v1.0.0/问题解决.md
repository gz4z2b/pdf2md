## 当前决策

| 问题 | 决策 |
| --- | --- |
| PDF 放在哪里 | 放在 `origin` 目录 |
| Markdown 源文件放在哪里 | 也放在 `origin` 目录 |
| Markdown 输出到哪里 | 输出到 `output` 目录 |
| 是否递归子目录 | 是 |
| 源文件本身是 Markdown | 不做转换，直接复制到 `output` 对应位置 |
| 是否导出图片 | 能提取的 PDF 图片导出到 `output/sources`，Markdown 中写入图片链接 |
| 使用什么 PDF 库 | PyMuPDF |
| 是否做网页界面 | v1.0.0 不做 |
| 是否做 OCR | v1.0.0 不做 |
| 已存在输出文件 | 默认直接替换为最新转换结果 |
| 页码标题 | 不再写入 `## 第 N 页` |
| 标题识别 | 单独成行、较像章节名且没有标点结尾的短中文行转为 Markdown 标题 |
| 序号识别 | `1、内容`、`(1) 内容`、行内 `(1)(2)` 形式尽量转为 Markdown 有序列表 |
| 表格识别 | 使用 PyMuPDF 表格识别能力，并过滤明显像正文排版的伪表格 |
| 长链接换行 | 尽量合并 PDF 提取时被拆开的 URL |
| 正文换行 | 日期、字段、横线和排行等结构行不乱粘；明显的短尾折行尽量合并 |
| 附件卡片 | 识别文件名和大小，输出为 `附件：文件名（大小）` |
| 图片目录 | 每个 PDF 的图片放在 `output/sources/源文件相对路径去后缀/`，避免不同文件重名 |
| `.note` 文件名 | 源文件名末尾是 `.note` 时，输出文件和资源目录去掉 `.note` |
| 孤立序号 | `1.`、`2.` 这类单独数字标记不和前面的标题粘连，而是和后面的文字组成列表项 |

## 发现的问题

1. 初始仓库没有源代码，所以首次 CodeGraph 索引中没有可分析的代码节点；实现完成后已重新索引。
2. README 原来只有空索引，缺少架构、技术栈、功能说明和版本文档入口。
3. README 中已引用 `docs/rule.md` 和 `docs/test.md`，但这两个文件原来不存在。
4. 需求中没有明确输出目录，本次文档约定为 `output`，便于用户查找转换结果。
5. 初版输出会加入页码标题，并把列表、标题、表格基本当普通文本处理；验收后已修正。
6. 用户实际源目录可能同时包含 PDF 和 Markdown；批量层已扩展为统一扫描 `.pdf` 和 `.md`。
7. PyMuPDF 对带边框或多栏排版的内容可能误判为表格；当前通过表头、行数、单元格长度和列表特征做保守过滤，优先保证正文可读。
8. PDF 文本块顺序和换行不稳定，同一份文档里既有真正分段，也有 PDF 自动折行；当前用日期、标签、URL、Top 排行、附件、短尾文字等启发式规则平衡可读性。
9. 只用图片占位符会丢失真实内容；当前改为优先导出图片文件，只有无法拿到图片数据时才退回占位符。
10. 语雀/笔记导出的 PDF 常见文件名是 `xxx.note.pdf`，直接改后缀会生成 `xxx.note.md`；当前统一去掉末尾 `.note`，让输出文件名更干净。
11. 有些 PDF 会把列表数字和列表内容拆成两个文字块，旧逻辑会把数字粘到前面的标题上；当前通过识别孤立序号解决这个问题。

## 后续改进建议

1. 真实 PDF 的排版差异很大，v1.0.0 完成后应使用用户实际 PDF 做一次手动验收。
2. 如果用户后续需要扫描版 PDF，需要单独规划 OCR 版本。
3. 如果用户后续需要提取 PDF 中的视频、音频或嵌入附件，需要单独规划多媒体对象的类型识别和命名规则。
